# План: TusaMap — карта тусовок Астаны (Telegram Web App)

## Суть продукта

- **Telegram-бот** с Web App (URL открывается внутри Telegram).
- **Авторизация через Telegram**: вход по аккаунту Telegram, привязка к TG-логину (user_id, username) — без паролей.
- **Карта Астаны** с точками мероприятий («тусовок»): **все тусовки отображаются на карте** маркерами по месту проведения; по клику — превью и переход к странице события.
- **Каталог мероприятий**: концерты, вечеринки, ивенты.
- **Покупка билетов**: банковская карта или Toncoins (через Telegram).
- **Создание мероприятий**: организаторы могут добавлять свои тусовки.

Проект **tusa-map** — только фронтенд (Angular). Бэкенд/API предполагаются отдельно или позже.

---

## 1. Технологии и интеграции

| Задача | Решение |
|--------|--------|
| Карта | **Leaflet** или **Mapbox** (карта Астаны, маркеры, кластеризация) |
| Telegram Web App | **@telegram-web-app** (инициализация, theme, viewport, MainButton, BackButton, готовность к TON Pay) |
| Стиль под Telegram | Подтягивать `theme_params` из Telegram (bg, text, accent и т.д.) |
| Оплата картой | Внешний платежный провайдер (API бэкенда): Stripe, CloudPayments, и т.п. |
| Оплата TON | Telegram Stars / TON через Bot API или отдельный TON payment flow (зависит от бэкенда) |
| Данные мероприятий | REST API бэкенда (список событий, детали, создание, билеты) |
| Авторизация | Telegram Web App `initData` → бэкенд верифицирует подпись, привязывает сессию к TG user_id/username |

Фронт только отображает UI, дергает API (с передачей initData или токена) и редиректит на оплату/Telegram-методы.

---

## 2. Структура приложения (страницы/экраны)

1. **Карта** — главный экран: карта Астаны, маркеры мероприятий, при клике — превью/переход к событию.
2. **Список мероприятий** — фильтры (дата, категория, бесплатно/платно), карточки событий.
3. **Страница мероприятия** — описание, дата/место, цена, кнопки «Купить билет» (карта / Ton).
4. **Оформление билета** — выбор способа оплаты (карта / Ton), редирект на оплату или Telegram Pay.
5. **Мои билеты** — список купленных билетов (по данным из API).
6. **Создать мероприятие** — форма для организаторов (название, дата, место, описание, цена, фото). Отправка на бэкенд.
7. **Профиль/настройки** — данные из Telegram (имя, username, фото), привязка к TG-логину, выход (если бэкенд поддерживает «логаут» сессии).

Роутинг: например `/`, `/events`, `/events/:id`, `/events/:id/buy`, `/my-tickets`, `/create-event`, `/profile`.

## 3. Интеграция Telegram Web App

- Скрипт: `https://telegram.org/js/telegram-web-app.js`.
- При старте приложения:
  - Инициализировать `window.Telegram.WebApp`.
  - Включить `expand()` для полноэкранного режима.
  - Подставить тему: цвета фона, текста, кнопок из `theme_params` / `color_scheme`.
  - Настроить MainButton / BackButton где нужно (например, «Купить» на экране билета).
- `initData` — передавать на бэкенд для верификации пользователя Telegram (логин/авторизация без пароля).

---

## 3.1. Авторизация через Telegram (привязка к TG-логину)

- **Вход**: пользователь открывает Web App из бота — он уже «в контексте» Telegram. Отдельной страницы логина нет: личность определяется по данным Telegram.
- **initData**: при каждом запросе к API (или при первом «логине») фронт отправляет `window.Telegram.WebApp.initData` на бэкенд.
- **Верификация на бэкенде**: бэкенд проверяет подпись initData (HMAC с Bot Token), извлекает `user` (id, first_name, last_name, username, language_code, photo_url). По `user.id` создаётся или находится аккаунт в БД и привязывается сессия/токен.
- **Привязка к TG-логину**: один аккаунт в TusaMap = один Telegram user_id. Билеты, созданные мероприятия, «Мои билеты» — всё привязано к этому пользователю.
- **На фронте**: после успешной верификации бэкенд может вернуть JWT или session cookie — дальше все запросы с этим токеном. В профиле показываем данные из Telegram (имя, @username, аватар из initData или запроса к API).
- **Если Web App открыт не из Telegram** (прямая ссылка в браузере): initData пустой — показываем сообщение «Откройте приложение из Telegram-бота» или редирект на бота.

## 4. Карта (отображение тусовок)

- **Тусовки отображаются на карте**: каждое мероприятие с координатами показывается маркером на карте Астаны; пользователь видит, где что проходит, и может перейти к деталям по клику.
- Центр: Астана (координаты по умолчанию).
- Слой: OpenStreetMap или Mapbox (нужен токен для стилей).
- Маркеры: события (тусовки) с координатами; при большом количестве — кластеризация (например, Leaflet.markercluster).
- Клик по маркеру: popup с названием + ссылка «Подробнее» → страница мероприятия.
- Опционально: геолокация пользователя, фильтры по дате/категории прямо на карте.

---

## 5. Мероприятия и билеты

- **Список событий**: запрос к API (GET /events с query-параметрами). Карточки: фото, название, дата, место, цена.
- **Детали**: GET /events/:id. Кнопки «Купить билет» → выбор способа оплаты.
- **Оплата картой**: фронт вызывает API бэкенда (создание платежа), получает URL/сессию и редирект на страницу оплаты провайдера или открывает iframe.
- **Оплата Ton**: через Telegram (например, Telegram Stars) или отдельный TON flow — по тому, что предоставит бэкенд. На фронте — кнопка и вызов соответствующего метода/редиректа.
- **Мои билеты**: GET /users/me/tickets (или аналог). Отображение списка с QR или кодом билета (если API отдаёт).

---

## 6. Создание мероприятий

- Форма: название, описание, дата/время, адрес/место (можно + координаты для карты), категория, цена (или «бесплатно»), обложка.
- Отправка: POST (или PUT) на API. Авторизация через `initData` Telegram.
- После успеха — редирект на карту или на созданное событие.

## 7. Приоритеты (в каком порядке делать)

| Приоритет | Что | Зачем |
|-----------|-----|--------|
| **P0** | Основа + Telegram Web App + роутинг | Без этого приложение не открывается в боте и нет навигации |
| **P0** | Авторизация через TG (initData, привязка к логину) | Все действия привязаны к пользователю; без авторизации нет «Мои билеты» и создания событий |
| **P0** | Карта с отображением тусовок | Главная фича: тусовки на карте Астаны |
| **P0** | Список мероприятий + страница мероприятия | Пользователь должен видеть события и переходить к деталям |
| **P1** | Интеграция с API (events, сессия) | Реальные данные вместо моков; без API нет масштаба |
| **P1** | Покупка билетов (карта / Ton) | Основной сценарий монетизации и пользы |
| **P1** | Мои билеты | Пользователь видит купленное и может показать QR/код |
| **P2** | Создание мероприятий | Организаторы наполняют карту контентом |
| **P2** | Полировка | Геолокация, фильтры на карте, уведомления — улучшение UX |

**Кратко по порядку:** сначала основа и Telegram (P0), потом карта и тусовки на ней (P0), потом список и карточка события (P0), затем авторизация (P0). После этого — API (P1), оплата и «Мои билеты» (P1), создание мероприятий (P2), полировка (P2).

## 9. Что нужно от тебя для старта

- **Бэкенд**: будет ли свой (Node/Python/другое) или пока только моки на фронте?
- **Домен и бот**: URL Web App (HTTPS), имя бота — чтобы сразу подставлять в конфиг.
- **Карта**: бесплатно OSM + Leaflet или Mapbox (тогда токен)?
- **Оплата**: какие провайдеры планируешь (Stripe, CloudPayments, другой) и как именно будет TON (Stars vs отдельный кошелёк)?

Когда скажешь «начинаем», идём по приоритетам: сначала **этапы 1–5 (P0)**, затем **6–8 (P1)**, потом **9–10 (P2)**.
